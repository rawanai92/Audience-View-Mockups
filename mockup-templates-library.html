What's new in this mockup

1
Source filter row in Conversations picker — new secondary filter under the category pills. Uses the existing origin field on WhatsappTemplateDto.
2
Borderless kebab next to the status chip — three clear dots, no box around it. Subtle until hover, where it picks up a soft tile. One action today: Delete. Scales to Edit / Duplicate later. Available on every card regardless of status, so users can clean up rejected / pending templates too.
3
Source filter row in Campaigns dropdown — same pills as Surface 1, inside the dropdown panel above the list. Consistent behaviour across both surfaces.
4
Per-item source pill + kebab in the Campaigns dropdown — source labelled as a small pill on each row (matches the house pill language). Same borderless kebab as Surface 1 for Delete.
5
Delete — allowed state — fires when the template is referenced only by past Sent messages, by composer state (ephemeral, not persisted), or by Draft campaigns. Short copy, two-line dialog. Calls out the draft-campaign behaviour (node marks itself unconfigured — existing UX) so the user knows what happens to in-flight work, without scare copy.
6
Delete — blocked state — fires when the template is referenced by:
· a Scheduled Conversations message (status=Scheduled with a future scheduledFor), or
· any non-Draft campaign (PendingApproval / Live / Scheduled / Expired / etc.).

The campaign backend hard-rejects all edits to non-Draft campaigns, so the user literally can't swap the template — we must block. Dialog lists every blocker with a direct "Open →" link. Draft campaigns don't trigger this state.

Composer fallback (separate from deletion, handled in the composer not this dialog): if the user has a template selected in the composer (ephemeral React state) and someone else deletes it, the next Send attempt shows an inline error — "This template was deleted. Pick another or create one." This isn't a deletion-time blocker because the selection isn't persisted server-side.

Historical Sent messages: not a blocker. Engineering note: the message body should be snapshotted at send time so historical conversation views don't break when a template is later deleted.

Open question for refinement (still): should Delete also remove the template from Meta's side, or only from our library? App-library only is the simpler v1; Meta-side cleanup can come later.
